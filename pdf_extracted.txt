PRODUCT REQUIREMENTS DOCUMENT 
(PRD) 
Project Title 
Adaptive Noise-Aware IR Spectroscopy Signal Processing & Functional Group 
Identification System 
 
1. Objective 
Develop a modular system that: 
●​ Ingests raw IR spectroscopy signals (CSV) 
●​ Identifies and classifies noise types 
●​ Applies adaptive filtering using VSSLMS 
●​ Compares processed signals with a reference IR database 
●​ Outputs top compound matches + functional groups 
●​ Visualizes all stages via a dashboard 
 
2. Problem Statement 
Traditional IR signal analysis suffers from: 
●​ Noise contamination (thermal, environmental, detector) 
●​ Static filtering methods (non-adaptive) 
●​ Limited interpretability (no confidence scoring or explainability) 
This system solves: 
●​ Dynamic noise handling 
●​ Adaptive filtering 
●​ Robust spectral matching 
●​ User-facing interpretability 

 
3. Scope 
In Scope 
●​ CSV-based signal ingestion 
●​ Multi-class noise classification 
●​ Adaptive LMS filtering (VSSLMS) 
●​ IR spectral database matching 
●​ Functional group extraction (via peaks) 
●​ Dashboard visualization 
Out of Scope 
●​ Real-time hardware sensor integration 
●​ 2D IR spectroscopy 
●​ Deep learning-based classification 
●​ Cloud deployment (optional future scope) 
 
4. System Overview 
High-Level Workflow 
CSV Input → Preprocessing → Noise Classification → Adaptive LMS Filter  
→ Compensation Engine → Spectral Matching → Functional Group Detection → Dashboard 
Output 
 
This aligns with the 6-module architecture defined in your system design . 
 
5. Functional Requirements 

5.1 Signal Ingestion 
●​ Upload CSV file 
●​ Auto-detect headers 
●​ Handle missing values (NaN interpolation) 
●​ Normalize signal amplitude 
 
5.2 Preprocessing 
●​ Compute FFT 
●​ Adaptive cutoff selection 
●​ Generate: 
○​ Estimated clean signal (x_est) 
○​ Noise estimate 
 
5.3 Noise Classification 
●​ Classify into: 
○​ Non-stationary 
○​ Detector 
○​ Environmental 
○​ Thermal 
○​ Electronic 
●​ Apply priority-based decision logic 
●​ Fallback mechanism for unknown noise 
 
5.4 Adaptive Filtering Engine 
●​ Implement VSSLMS algorithm 
●​ Dynamic step-size (μ) adjustment 
●​ Stability constraint enforcement 
●​ Divergence detection and reset 
●​ Runtime variance monitoring 
 

5.5 Compensation Engine 
●​ Select filter parameters (M, μ) 
●​ Adjust based on signal power 
●​ Re-evaluate during runtime 
●​ Post-filter SNR validation 
●​ Trigger fallback if SNR < 0 dB 
 
5.6 Spectral Matching 
●​ Normalize signals 
●​ Resample database signals 
●​ Compute: 
○​ Pearson correlation 
○​ Spectral Angle Mapper (SAM) 
●​ Generate top-3 matches with scores 
 
5.7 Functional Group Detection 
●​ Extract peaks from filtered signal 
●​ Map peaks to known IR frequency bands 
●​ Output detected functional groups 
 
5.8 Dashboard (React UI) 
Display Components: 
●​ Raw vs Filtered Signal Graph 
●​ Noise Classification Result 
●​ Filter Parameters (M, μ) 
●​ SNR Improvement 
●​ Top-3 Compound Matches 
●​ Functional Groups Identified 
 

6. Non-Functional Requirements 
Performance 
●​ Processing time < 5 seconds for typical signals (N ≤ 10,000) 
●​ Efficient MATLAB execution 
Scalability 
●​ Modular design allows independent upgrades 
●​ Database expansion supported 
Reliability 
●​ Fallback mechanisms: 
○​ Noise classification fallback 
○​ Filter rerun on poor SNR 
Usability 
●​ Simple CSV upload interface 
●​ Clear visualization of results 
 
7. System Architecture 
Frontend 
●​ React.js 
●​ Charting library (Chart.js / Recharts) 
Backend (Middleware) 
●​ Python (FastAPI preferred) 
●​ Handles: 
○​ File upload 
○​ MATLAB execution 

○​ JSON response 
Processing Engine 
●​ MATLAB 
●​ Executes Modules 1–6 
 
8. Data Flow 
React UI → API → MATLAB Engine → Processing → JSON Output → UI Rendering 
 
 
9. Data Model 
Input 
{ 
  "signal": [float], 
  "sampling_rate": float 
} 
 
Output 
{ 
  "noise_type": "Environmental", 
  "snr_before": float, 
  "snr_after": float, 
  "filter_params": { 
    "M": int, 
    "mu": float 
  }, 
  "top_matches": [ 
    {"name": "Ethanol", "score": 0.87}, 
    {"name": "Methanol", "score": 0.82} 
  ], 
  "functional_groups": ["O-H", "C-H"] 
} 

 
 
10. User Stories 
Student/User 
●​ Upload a noisy IR signal 
●​ View cleaned signal 
●​ Understand noise type 
●​ Identify compounds and functional groups 
Evaluator 
●​ Validate algorithm robustness 
●​ Inspect filtering performance 
●​ Compare results with known references 
 
11. Success Metrics 
●​ ≥ 85% accuracy in compound matching (on test dataset) 
●​ Positive SNR improvement in ≥ 90% cases 
●​ Correct noise classification in ≥ 80% cases 
●​ Functional group detection accuracy ≥ 75% 
 
12. Risks & Mitigation

Tunable thresholds 
LMS divergence 
Runtime stability constraint

Integration failure 
API abstraction layer

This is not just a filtering tool. It is: 
An intelligent IR signal analysis platform combining adaptive DSP, noise 
intelligence, and chemical interpretation.

Deliver a high-fidelity, real-time analytical dashboard for IR signal processing that: 
●​ Visualizes adaptive filtering behavior 
●​ Provides interpretable signal metrics 
●​ Displays compound matching results 
●​ Highlights detected functional groups 
●​ Enables parameter control for experimentation

●​ Signal-first visualization (data dominates UI) 
●​ Low-latency perception (fast feedback loops) 
●​ Technical aesthetic (lab-grade interface) 
●​ Minimal cognitive load 
●​ Explainability over decoration

●​ Pearson score 
●​ SAM score

●​ SNR < 0 
●​ No match found 
👉 Add: 
●​ Warning banner

================================ TECHSTACK.pdf ================================
 
1. Recommended Tech Stack (Best Fit) 
Core Architecture 
React (Frontend) 
      ↓ 
FastAPI (Backend / Orchestration Layer) 
      ↓ 
MATLAB Engine API (Processing Core) 
 
 
2. Why This Stack Wins 
Frontend — React (Keep it) 
●​ Strong ecosystem for dashboards 
●​ Excellent charting libraries (Recharts, Chart.js) 
●​ Component-based → easy modular UI 
Verdict: Correct choice. No change needed. 
 
Backend — FastAPI (Python) 
This is the critical decision layer. 
Why FastAPI over Node.js: 
●​ Native integration with MATLAB Engine for Python 
●​ Handles numerical data (arrays, signals) efficiently 
●​ Async support → faster API responses 
●​ Built-in validation (Pydantic) 
👉 Node.js + MATLAB = friction​
👉 Python + MATLAB = native synergy 

Verdict: 
FastAPI is the most efficient orchestration layer for your system. 
 
Processing Layer — MATLAB 
●​ Runs: 
○​ LMS filter 
○​ Noise classification 
○​ Compensation engine 
○​ Matching logic 
Integration Method: 
Use: 
MATLAB Engine API for Python 
 
Example flow: 
import matlab.engine 
eng = matlab.engine.start_matlab() 
result = eng.adaptive_ir_filter_v2(signal) 
 
 
3. Supporting Components (Essential) 
Data Exchange Format — JSON 
●​ Lightweight 
●​ Easy React integration 
●​ Supports structured outputs (matches, parameters) 
 
File Handling 
●​ Upload CSV → FastAPI 

●​ Convert → NumPy array → MATLAB 
 
Charting (Frontend) 
Use: 
●​ Recharts (clean, simple)​
OR 
●​ Plotly.js (if you want advanced spectral plots) 
 
4. Optional but High-Value Additions 
A. WebSocket (Real-time updates) 
If you want: 
●​ Live LMS convergence graph 
●​ Step-size (μ) evolution 
Use: 
●​ FastAPI WebSockets 
 
B. Database (For IR Reference Data) 
Option 1 (Simple): 
●​ JSON / CSV (local) 
Option 2 (Better): 
●​ SQLite 
Option 3 (Scalable): 
●​ PostgreSQL 

👉 Start simple. Upgrade later.

6. Architecture Decision (Blunt Reality) 
❌ Avoid: 
●​ Node.js backend (poor MATLAB integration) 
●​ Direct MATLAB → frontend (no control layer) 
●​ Monolithic MATLAB UI (not scalable) 
✅ Do: 
●​ Use Python as control plane 
●​ Keep MATLAB as compute engine 
●​ Keep React as presentation layer 
 
7. Execution Workflow 

1. User uploads CSV (React) 
2. FastAPI receives file 
3. Converts → NumPy array 
4. Sends to MATLAB Engine 
5. MATLAB processes: 
   - Filtering 
   - Classification 
   - Matching 
6. Returns structured output 
7. FastAPI → JSON response 
8. React renders dashboard 
 
 
8. Risk Considerations 
MATLAB Engine Bottleneck 
●​ MATLAB is not inherently scalable 
●​ Each request may spawn/hold session 
Mitigation: 
●​ Maintain a persistent MATLAB session 
●​ Queue requests if needed 
 
Latency 
●​ MATLAB execution can be slow for large N 
Mitigation: 
●​ Preprocess in Python where possible 
●​ Optimize MATLAB code (vectorization) 
 
9. Final Recommendation 

React + FastAPI + MATLAB Engine is the optimal stack. 
This gives you: 
●​ Clean separation of concerns 
●​ Minimal integration complexity 
●​ Strong academic + industry credibility
